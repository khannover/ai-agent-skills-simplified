# Skill: Planning & Task Breakdown

**Use when:** You have a spec and need to figure out the implementation order.

## When This Applies
- After a spec is confirmed
- Before any code is written
- When a task feels large or uncertain

## Process

1. **List all tasks** — everything needed to satisfy the spec
2. **Make tasks atomic** — each task should be completable in one focused session
3. **Define acceptance criteria** — each task has a testable "done" condition
4. **Order by dependency** — what must be true before each task can start?
5. **Flag unknowns** — mark any task where you're unsure of the approach

## Good Task Format

- [ ] Task: [verb] [noun]
  - Acceptance: [how you'll know it's done]
  - Depends on: [prior task or "nothing"]

## Exit Criteria (before moving to BUILD)
- [ ] Every task has an acceptance criterion
- [ ] Tasks are ordered by dependency
- [ ] No task is "implement everything" — each is one slice
